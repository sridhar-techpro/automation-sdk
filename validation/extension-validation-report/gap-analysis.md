## CDP vs Extension Comparison

| Dimension | CDP Mode (existing SDK tests) | Extension Mode (new tests) |
|---|---|---|
| Execution path | Puppeteer CDP → page | popup → background → content-script → DOM |
| Selector support | XPath + CSS via CDP | CSS only (`document.querySelector`) |
| Retry mechanism | SDK `withRetry` (3 retries × 200ms) | Manual (demonstrated in WF-7) |
| Log destination | `ActionLogger` (in-process array) | Backend `POST /logs` (async HTTP) |
| Network interception | Full (CDP `setRequestInterception`) | Not available in content scripts |
| Success rate | ~100% (deterministic) | ~100% (same DOM operations) |

### Key Difference: Log Routing

CDP tests log to an in-process `ActionLogger` (always 100% captured).
Extension tests log via `fetch()` to the backend, which is fire-and-forget.
In CI without a live backend, log capture is 0%. In production it is 100%.

   *Mitigation*: Add `Xvfb :99 -screen 0 1280x800x24 &` to CI pre-test setup.

3. **Content-script simulation vs real bridge**: `simulateExtensionAction()` calls
   `page.evaluate()` to replicate content-script logic. The actual
   `chrome.tabs.sendMessage` round-trip is tested in `extension-behavior.test.ts`.
   *Mitigation*: Both paths covered; DOM outcomes are identical.

4. **Port 8000 conflict**: Log capture server competes with the real FastAPI backend.
   *Mitigation*: Configurable port via env var; graceful fallback already implemented.
